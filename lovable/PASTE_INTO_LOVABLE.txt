Build a premium medical evidence explorer web app called "Node-Positive MIBC Evidence Explorer".

=== SITE SCOPE (CRITICAL — READ FIRST) ===

This website is for ONE patient category only:

  REGIONALLY NODE-POSITIVE MIBC: cN1–cN3, cM0 (no distant metastases)
  Treatment intent: systemic therapy first, then possible curative local treatment (cystectomy or radiation)

It is NOT a general bladder cancer site. Do NOT build navigation or content for:
- Localized MIBC without nodes (cN0 M0)
- Bladder-preservation / trimodality therapy as primary focus
- Post-cystectomy adjuvant-only pathways (except as related context)
- Recurrent or metastatic (M1) first-line treatment (except clearly labeled cross-setting trials like EV-302)

Every page must reinforce scope:
- Persistent header subtitle: "Regionally node-positive MIBC · cN1–N3 M0 · systemic therapy before surgery"
- Homepage hero: "For patients with muscle-invasive bladder cancer AND involved pelvic lymph nodes, but no distant spread"
- Footer + disclaimer: "This tool does not cover localized (node-negative) MIBC, bladder-sparing protocols, or metastatic disease."

Example profile within this category: pT2 high-grade, L1, PET-positive external iliac node, cN1 cM0 — perioperative systemic therapy before possible radical cystectomy.

Dataset version 1.2 — 20 studies scored for node-positive / perioperative relevance.

=== REVIEW QUEUE PAGE (REQUIRED — human approve/reject) ===

Add nav item: **Review queue** (badge with pending count from getAgentStatus().counts.pending).

This page is where the user decides which auto-discovered studies to INCLUDE in the evidence set.
Scope reminder at top: cN+ M0 perioperative MIBC only.

On load and after every action: call fetchReviewQueue() from agentApi.ts (GET /api/pending).

Each queue item card shows:
- title (item.title)
- relevance_score badge
- source badge (pubmed / clinicaltrials.gov)
- nct_id or pmid if present
- enrollment n if present
- review.scope_verdict (assessment sentence)
- review.pros as green bullets, review.cons as amber bullets
- link to item.url (open in new tab)
- Left border color by review.suggestion: strong_candidate/worth_review=green, mixed=amber, likely_exclude=slate

Two buttons per card:
- **Approve — include** → approvePaper(item.id) → toast "Approved" → remove from queue / show green "Included"
- **Reject — exclude** → optional reason dialog → rejectPaper(item.id, reason) → toast "Rejected" → remove from queue

Empty state: "No items pending — run Refresh evidence on Overview to search PubMed + ClinicalTrials.gov"

Show live connection required banner if API unreachable (same as Overview).

API:
- GET /api/pending → array with title, url, review { scope_verdict, suggestion, pros, cons }, relevance_score
- POST /api/approve {"paper_id":"..."}
- POST /api/reject {"paper_id":"...", "reason":"optional"}

Note: Approve marks paper trusted in papers_index.json on the user's PC. Does not auto-add to trials.json yet.

Fallback: if VITE_AGENT_API_URL is unset, use static JSON imports from src/data/ and show banner "Connect live agent — set VITE_AGENT_API_URL".

API endpoints (header X-API-Key on all except /api/health):
- GET /api/health, /api/status, /api/trials, /api/papers, /api/pending
- POST /api/search, /api/refresh, /api/approve {"paper_id":"..."}, /api/reject {"paper_id":"...", "reason":"..."}
